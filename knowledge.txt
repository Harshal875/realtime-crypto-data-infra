8) How does Kafka decide which slot a message goes into?

producer.produce(
    topic=KAFKA_TOPIC,
    key=symbol_name.encode(),   # ← "BTCUSDT", "ETHUSDT", or "SOLUSDT"
    value=json.dumps(message).encode(),
)

Kafka takes the key ("BTCUSDT"), runs it through a hash function (like a formula that converts a word into a number), 
and uses that number to pick the slot.

suppose in the incoming stream of data, theres a message like 4.5523 BTCUSDT, the key hashes the BTCUSDT part
and converts into a number, lets say 0, and therefore the message goes into slot 0

slot 0 (BTC): tick#1, tick#2, tick#3, tick#4 ...  ← always in order
slot 1 (ETH): tick#1, tick#2, tick#3 ...           ← always in order
slot 2 (SOL): tick#1, tick#2 ...                   ← always in order


9) Offset = the serial number Kafka assigns to each message

suppose the topic has 5 messages, and now consumer starts reading from topic, it first reads the 5 fastly then catches the pace with real time, 
but now suppose consumer stopped at offset 50, but messages keep on coming in topic till 100, now suppose consumer starts again, will it read from 100

Here's why: Kafka remembers where each consumer group left off

Consumer reads offset 0, 1, 2 ... 50
Consumer crashes / stops
Messages 51, 52 ... 100 arrive in Kafka (Kafka stores them, doesn't care)

Consumer restarts
Consumer asks Kafka: "where did analytics-service stop?"
Kafka says: "offset 50"
Consumer reads 51, 52, 53 ... catches up to 100, then goes real-time

No messages lost. No messages skipped.

10) Zookeper:

Imagine Kafka is a big office with multiple people (brokers) working in it.

Now someone needs to answer these questions at all times:

Which broker is currently the "leader" (main one in charge)?
Which consumer group has read up to which offset?
Is broker #2 still alive or did it crash?
Zookeeper is the office manager who tracks all of this. It's a separate process whose only job is to watch Kafka and keep track of that metadata.

In our project:

Kafka cannot start without Zookeeper in our setup — that's why in docker-compose.yml you see:
Kafka waits for Zookeeper to be healthy before it even starts.


11) the topic messages are stored on disk, but wont the retreival on disk be slow by consumer when compared to RAM

Random access (slow) — imagine a librarian who has to walk to a random shelf, find a random book, open to a random page every time.

Sequential access (fast) — imagine a librarian who only ever adds to the end of one shelf, and readers only ever read from where they left off, going forward.

Kafka only does the second one:

Producer always appends to the end of the file. Never inserts in the middle.
Consumer always reads forward from its last offset. Never jumps around.
Sequential disk reads/writes are so fast that they can actually match RAM speed in practice.

### THREADING , LOCKS:

1) now the server.py of OrderBookService has to do two completely unrelated things simultaneously:

Kafka consumer — keep reading from Kafka topic, update the latest prices in a dict
gRPC server — answer incoming requests (function calls) from other services

these cant take turns , both have to be done simultaneously, so the code runs 2 threads:

Thread 1 (background):        Thread 2 (main):
kafka_consumer_thread()        gRPC server on port 50051
  ↓                              ↓
reads Kafka forever            listens for incoming calls
  ↓                              ↓
updates order_books dict       reads order_books dict
                               returns answer


2) Two threads are both touching the same order_books dict:

Thread 1 is writing to it (new price from Kafka)
Thread 2 is reading from it (answering a gRPC request)


What if Thread 2 reads the dict exactly in the middle of Thread 1 writing to it? 
Thread 1 has written the new bid but hasn't written the new ask yet. Thread 2 reads a half-updated state — new bid, old ask. 
Wrong spread. Corrupted data.


3) The fix is threading.Lock():

lock = threading.Lock()

# Thread 1 writing:
with lock:
    order_books["BTCUSDT"] = { "bid": 81500.00, "ask": 81500.01, ... }

# Thread 2 reading:
with lock:
    return dict(order_books["BTCUSDT"])

Only one thread can hold the lock at a time. So you'll never get a half-written read.

3) Part 2: Postgres and TimescaleDB — what's the difference?
You probably know Postgres — it's a relational database. Tables, rows, SQL. Like a very powerful Excel that lives on a server.

TimescaleDB is just a plugin on top of Postgres that makes it dramatically better for time-series data. You install it, and then your normal Postgres tables can become "hypertables."

What's a hypertable?

Imagine your price_ticks table eventually has 1 billion rows — one row per tick, months of data.
If you query WHERE time > last hour, without TimescaleDB, Postgres scans all 1 billion rows to find the relevant ones.

With TimescaleDB, it automatically splits your table into time-based chunks behind the scenes:
price_ticks table (what you see — looks like one table)

Behind the scenes:
  chunk_2026_05_01  ← all ticks from May 1
  chunk_2026_05_02  ← all ticks from May 2
  chunk_2026_05_03  ← all ticks from May 3
  ...

When you query WHERE time > last hour, TimescaleDB knows exactly which chunk to look at. It doesn't touch the other chunks at all. 
Query goes from scanning 1 billion rows to scanning maybe 50,000 rows.

You still write normal SQL. TimescaleDB handles the partitioning silently. 
That's why it's used by every serious time-series application — stock data, IoT sensor data, metrics, logs.

In init.sql:

This one line turns a normal Postgres table into a hypertable. That's all you need.

5) Part 4: What is OHLCV?
OHLCV stands for: Open, High, Low, Close, Volume.

For one minute of BTC trading:
Open  = the very first price at the start of the minute   → 81456.78
High  = the highest price reached that minute             → 81502.00
Low   = the lowest price reached that minute              → 81430.00
Close = the very last price at the end of the minute      → 81498.50
Volume= total BTC quantity that changed hands that minute → 12.5 BTC

On a candlestick chart:

If Close > Open → green candle (price went up that minute)
If Close < Open → red candle (price went down that minute)

Every chart you've ever seen on Groww, Zerodha, TradingView — every single candle is computed exactly this way.



6) Part 5: What is VWAP?
VWAP = Volume Weighted Average Price.

The regular average price for a minute: just add up all prices and divide by count. But this treats a 0.001 BTC trade the same as a 50 BTC trade. That's not accurate.

VWAP weights each price by how much volume happened at that price:

VWAP= ∑(price×quantity) / ∑quantity 

In plain English: "what was the average price, but bigger trades count more?"
tick 1: price=81456, qty=0.5  →  price × qty = 40,728
tick 2: price=81500, qty=10   →  price × qty = 815,000
tick 3: price=81480, qty=2    →  price × qty = 162,960

VWAP = (40,728 + 815,000 + 162,960) / (0.5 + 10 + 2)
     = 1,018,688 / 12.5
     = 81,495.04

Example:

The tick 2 with 10 BTC pulled the average much closer to 81500 than the simple average would have. That's more accurate — it reflects where most of the actual money moved.

Why traders care about VWAP: It's used as a benchmark. If you bought BTC at 81,490 and VWAP for the day is 81,495, you got a slightly below-average price — good. 
Institutional traders specifically try to execute orders near VWAP to minimize market impact.

In your code, the accumulator keeps a running pv_sum (price × qty sum) and vol_sum (qty sum). At the end of each minute:
vwap = pv_sum / vol_sum

8) Elasticsearch — what is it and why is it here alongside Postgres?
You already have Postgres storing every tick. Why also put it in Elasticsearch?

Postgres is great at: structured queries, time ranges, aggregations, JOINs.

Elasticsearch is great at: full-text search, fuzzy matching, complex filters across many fields simultaneously, and it returns results in milliseconds even across hundreds of millions of documents.

VERY IMP: In your GraphQL service (Week 6), 'searchTicks' query reads from ElasticSearch and  lets users do things like:
"Give me all BTCUSDT ticks where bid was between $80,000 and $82,000"

That kind of multi-field range query across millions of rows is what Elasticsearch is purpose-built for. Postgres can do it too, but Elasticsearch does it faster for search-heavy workloads.

Think of it this way:

Postgres = your permanent ledger (every tick, forever, structured queries)
Elasticsearch = your search engine (fast lookups, filters, the thing that powers the search bar)



9) Part 8: Batch commits — why not write to Postgres tick by tick?
You might notice this in the code:

Why not just commit after every single tick?

Each commit() call involves:

Writing to the Postgres transaction log on disk
Network round-trip between your code and Postgres
That takes ~2ms. At 60 ticks/second, committing every tick = 120ms per second spent just on commits. Your service can't keep up with the incoming data.

Batching 50 ticks together = 1 commit per 50 ticks instead of 50 commits. 50x less overhead. The data is still safe — it's in RAM in an open transaction while you collect 50 ticks, then all 50 hit disk in one shot.
